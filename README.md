# WME OpenMaps (Candy Remix)

A modernized, performance-focused fork of the original [WME OpenMaps](https://github.com/Glodenox/wme-om) script by Tom 'Glodenox' Puttemans. 

This userscript augments the [Waze Map Editor](https://www.waze.com/editor) by allowing editors to overlay external, open-data maps (such as local cadasters, high-resolution orthophotos, and administrative boundaries) directly onto the editing canvas.

The **Candy Remix** focuses on aligning the script with modern WME SDK standards, improving UI/UX density, and adding advanced cartographic tools for power editors.

## Key Features

* **Advanced Visual Adjustments:** Take full control of how layers render with native CSS blend modes (Multiply, Screen, Overlay, etc.), alongside real-time sliders for Opacity, Brightness, Contrast, Saturation, and Gamma.
* **Modernized UI:** A rebuilt sidebar interface featuring clean metadata headers, easily comparable monospace bounding boxes, and compact avatar badges with regional flags.
* **WME SDK Native:** Rewritten to utilize the official Waze Map Editor SDK, ensuring maximum stability and future-proofing against internal Waze rendering updates.
* **Asynchronous Engine:** Background "Terms of Use" validation and map layer state synchronization are handled asynchronously to ensure the editor remains snappy and responsive.

## Changelog

### 2026.04.04

- **Fix (satellite + Google My Maps, v2026.04.04.1):** Restored **v37-style hook gating** — `syncOpenMapsLayerIndices` no longer installs global OpenLayers **`Layer.setZIndex`** / **`resetLayersZIndex`** hooks on every run (the v40 rollback had put that back), so satellite is not wrapped whenever any OpenMaps row exists. Hooks attach when **`pinOpenMapsOverlayStackTop`** actually pins ESRI/inspector overlays; **`addlayer` / debounced `moveend`** pin listeners register only when **`openMapsOverlayPinStackHasWork()`**. **`minForeignAbove`** for tile stacking now walks **`olMap.layers`** with **resolved OpenLayers refs** so My Maps vectors are not misclassified as foreign when `W.map.getLayers()` uses different object identity than `handles[].layer`.
- **Fix (satellite + Google My Maps, v2026.04.04.2):** Tile stacking math now uses **`olMap.getLayerIndex`** consistently for the **aerial floor** and the **lowest foreign layer above aerial** (avoids mixing `layers[]` enumeration index with stack order). Google My Maps vectors are **`removeLayer`’d** from the map when the row should not display (hidden, no active sub-layer, out of area, or ToU not accepted), not left on the stack as invisible OL layers. KML fetch no longer runs **`syncOpenMapsLayerIndices`** after every load.
- **Diagnostics (Google My Maps, v2026.04.04.3):** Opt-in troubleshooting: set `localStorage.setItem('openmaps-gmm-diag','1')` and reload WME. OpenMaps then logs **`olMap` layer stack** snapshots around sync / GMM map ops / KML load, logs any **`setLayerIndex`** or **`removeLayer`** on **non-OpenMaps** layers, and warns if **`olMap.layers.length`** changes during **`syncOpenMapsLayerIndices`** (to debug native layers such as roads disappearing). Disable with `localStorage.removeItem('openmaps-gmm-diag')` and reload.
- **Fix (satellite + Google My Maps, v2026.04.04.4 - .6):** WME satellite maps turning blank (tile requests dropped/blocked) while roads remain visible indicates the OpenLayers `moveend` event chain or the native Waze map state synchronizer is being broken. OpenMaps now **bypasses WME’s `W.map.addLayer` wrapper entirely** for heavy vector layers (My Maps / ESRI_FEATURE), adding them directly to the underlying OpenLayers engine. This stops WME’s React state from tracking thousands of foreign vectors. Hidden My Maps now also **lazily defer KML fetching and map attachment** until they are checked, preventing rapid attach/detach sequences or blocking `XMLHttpRequest` calls from starving Waze's tile renderer on boot. Event registrations are fully decoupled from Waze's move lifecycle. KML restricts loading to **1500** features maximum.

### 2026.04.03

- **Fix (satellite + Google My Maps, v2026.04.03.34):** With only a My Map (or other vector overlay) active, constant OpenLayers stack re-pinning on every pan/zoom could stop WME satellite imagery from loading. ESRI_FEATURE / My Maps vectors are pinned only when the row is on and the layer is visible; hidden maps no longer re-enter the stack via the layer sweep; pan/zoom re-pin is debounced (~220 ms); overlay z-index rules clear when nothing is pinned; tile stacking also keys off the `satellite_imagery` layer name when `earthengine-legacy` is missing.
- **Fix (satellite + Google My Maps, v2026.04.03.35):** My Maps KML is **excluded** from the ESRI-style overlay pin (top-of-`olMap` index, `!important` z-index, hoisting layer `div`s in the viewport). It is ordered only through **`syncOpenMapsLayerIndices`** above aerial imagery, like tile overlays. The pin stack could still break satellite tiles when My Maps was the sole overlay.
- **Fix (satellite + Google My Maps, v2026.04.03.36):** While a My Map row is **hidden** (or out of area / OL layer not visible), **`syncOpenMapsLayerIndices` no longer calls `setLayerIndex` on that vector** — it remained on the map and was still being reordered. Stacking math uses only participating handles. The **`setZIndex` / `resetLayersZIndex` → overlay stylesheet** path runs only when **ESRI_FEATURE** pin work or **Map Inspector** overlays exist, so satellite layers are not pulled through that pipeline on every pan/zoom when only a hidden My Map sits in Active Maps.
- **Fix (satellite + Google My Maps, v2026.04.03.37):** **`syncOpenMapsLayerIndices` no longer installs** the global OpenLayers **`Layer.setZIndex`** / **`Map.resetLayersZIndex`** hooks on every sync. Those hooks were still attached as soon as any OpenMaps row existed (e.g. a hidden My Map), so **every** WME layer—including satellite—went through our wrapper. Hooks and **`moveend` re-pin** are registered only when **ESRI_FEATURE** pin work or **Map Inspector** overlays need them; **`pinOpenMapsOverlayStackTop`** installs the same hooks the first time it pins a non-empty stack.
- **Fix (satellite + Google My Maps, v2026.04.03.38):** Hidden/out-of-area My Maps are now **fully detached from the map** (`W.map`/`olMap`) instead of staying as invisible vectors. This prevents “My Map present in list” from touching OpenLayers layer state and satellite tile flow. The My Maps vector is recreated only when that row becomes visible again.
- **Hotfix (startup, v2026.04.03.39):** Reverted the `v2026.04.03.38` My Maps detach path after startup regressions. The script keeps `v2026.04.03.37` behavior as stable baseline while satellite/My Maps debugging continues.
- **Hotfix rollback (startup, v2026.04.03.40):** Stepped back one more level to the **v2026.04.03.36 runtime path** (restore hook/`moveend` registration flow from before `v2026.04.03.37`) to recover script startup reliability first.
- **Startup fix (v2026.04.03.41):** In some bridge/Tampermonkey environments, `wme-initialized` appears but `wme-ready` never fires. OpenMaps now keeps the normal `wme-ready` listener **and** sets a guarded fallback timer (single launch) so startup does not stall at “waiting for wme-ready signal…”.
- **Fix (satellite + Google My Maps, v2026.04.03.42):** Google My Maps vectors are **never** passed through `syncOpenMapsLayerIndices` `setLayerIndex` (previously only hidden/out-of-area/invisible rows were skipped). Visible KML was still reordered above aerial on every sync and KML refresh, which could stop satellite tiles after pan/zoom. My Maps stays at the default OpenLayers stack position; tile/WMS/ESRI overlays still sync above aerial.
- **Your maps (Google My Maps):** Paste the **normal browser URL** from Google My Maps while **editing, previewing, or sharing** (e.g. `…/maps/d/edit?mid=…`); you do **not** need a separate “KML link”—OpenMaps reads the `mid` and loads geometry via Tampermonkey. Parser trims stray quotes/punctuation; hash/query fallbacks. Entries persist in `userMaps`; accept **Google’s terms** per map. Same list is intended for future **WMS / ArcGIS / XYZ** custom URLs.
- **WMS (ArcGIS REST):** Map details include a checkbox to turn **REST viewport feature queries** (Map Inspector `/query` traffic) on or off **separately** from WMS tiles. Tiles still follow Active Maps visibility and per-layer toggles; when probe is off, inspector no longer runs those REST requests. Queries are gated by ToU, out-of-area, and whether the map is disabled in Active Maps, and no longer require the WMS tile layer’s OpenLayers visibility (so brief desyncs or missing layer handles are less likely to block `/query` alone).
- **Region Universal (`UN`):** The “draw boundary box on map” control is hidden; saved bbox state is cleared and the bbox vector overlay stays off (world-scale bounds are not useful in the editor).
- **ESRI_FEATURE (FeatureServer vectors):** Map options no longer show the WMS-style transparent checkbox, CSS filter sliders / blend / invert / reset, the pixel-manipulations block, or **opacity**—vectors are always drawn at full opacity in-engine; `applyFilters` already skipped vectors. Non-`UN` maps still get the bbox checkbox when applicable.
- **Map Inspector:** The per-map checkbox in the feature list only **expands or collapses** rows under that map. It no longer toggles Active Maps visibility (the sidebar eye / layer checkbox still controls whether the overlay is on), so collapsing a group no longer disables the map or removes it from the inspector for that reason alone.
- **UI:** Hide the “Search this area” Places pill (**WME React**: `container--*`, `iconContainer--*`, `.w-icon-search`, `div.text` — not Google `.gm-style`) with `:has(...)` CSS and a `div.text` phrase sweep; legacy Google-styled map controls are still hidden where they appear. Sweeps run **immediately** on mutation (plus rAF / debounced follow-up) and again on map move/zoom end so the chip does not flash after the first pan post-reload.
- **Map Inspector:** Text in the feature detail callout (attributes / description) can be selected and copied; the map canvas uses `user-select: none` for panning, which previously prevented selection inside the callout.
- **Map Inspector:** Feature detail callout stays open while panning the map (OpenLayers `move` is treated as a pan, so dismiss-on-click-outside no longer misfires when pointer-move events are not delivered during drag). The floating inspector window can be dragged partly off-screen (a 40px strip stays reachable to grab it again). The feature list is the only vertically scrolling region inside the inspector (the outer window body no longer scrolls as a whole). The list uses `overflow-x: hidden` and tighter row/map-head layout so a horizontal scrollbar does not appear.
- **Fix:** The feature list had collapsed to zero height (`flex: 1 1 0` inside an auto-sized column). The list now uses `flex: 1 1 auto`, a minimum height, and a `max-height` cap so rows stay visible and scroll inside the list. Un-minimizing the inspector restores the window body with `display: flex` (not `block`) so the column layout keeps working.
- **Map Inspector / ESRI_FEATURE:** Hover and selection no longer change **whole** FeatureServer layer order (which felt like shifting the whole map stack). Inspector stacking is limited to overlay layers: for ESRI points, temporary **lift** vector layers draw a **cloned** symbol (same avatar circle style as the source) so that one feature reads on top of other overlays; white halos sit under the rings; `requestAnimationFrame` re-pins overlays after new layers attach. Global ESRI_FEATURE vs WME POI stacking is still only what `syncOpenMapsLayerIndices` applies (not per-hover).
- **Map Inspector / WMS & WFS:** Rows that only have stored geometry (no live ESRI vector feature ref) now use the same avatar-colored point lift/halo/ring and the same line/polygon outline styling as ESRI_FEATURE highlights, driven from the row’s map title color.
- **Map Inspector:** Those geometry-only symbols stay on the map whenever the inspector window is open (subdued line/polygon styling; points match list avatar circles). Hover/selection overlays still draw on top; closing the inspector clears the persistent symbols.
- **Map Inspector:** Pointer hover and map click use the same top-down vector hit-testing as ESRI_FEATURE layers for those geometry-only symbols (including a pixel-radius fallback for small points).
- **Map Inspector:** Map clicks for inspector symbols and ESRI_FEATURE points are handled in **document capture** phase (when the pointer is over the map canvas) so WME **Places** and similar overlays that sit above the map in the DOM no longer eat the click before OpenLayers runs.
- **Map Inspector / stacking:** Inspector highlights, viewport-geometry symbols, and ESRI_FEATURE vectors are pinned in **one** OpenLayers pass (frontmost: rings/halos/lifts → persistent geom → each ESRI bbox then main). An `addlayer` hook re-pins after WME adds layers (e.g. Places) so symbols stay **drawn** above those overlays, not only hit-tested.
- **Map Inspector / Places:** After pinning, the script sets **CSS `z-index`** on those OpenLayers layer `div`s (and **mirrors** it on `canvas` / `svg` under the div with `!important` where needed). Values are assigned **from 2147483647 downward** (one step per layer in the ordered stack) so the **backmost** layer — the main ESRI_FEATURE vector with all default symbols — is **not** stuck at “`base` + 1” while WME Places uses a z-index **between** that and the inspector lift/halo layers (which made hover/selection clones look fine but left normal symbols visually under Places). The overlay stack is re-pinned after FeatureServer features load so stacking survives redraws.
- **Map Inspector / Places (WME layer refs):** `olMap.layers.indexOf(handle.layer)` can be **−1** when WME wraps or replaces the `OpenLayers.Layer` instance. The script now resolves layers by **`layer.name`** before `setLayerIndex` and before applying `z-index`, so pinning and styling hit the **live** map layer (otherwise changes had no visible effect while inspector-only layers still worked).
- **Map Inspector / Places (ESRI_FEATURE):** FeatureServer vector layers no longer use `map.title` as the OpenLayers `layer.name` (duplicate titles across catalog rows made name-based resolution pick the wrong layer). Each ESRI_FEATURE layer now uses a unique name `OpenMaps_ESRI_FEATURE_<map id>` and `openMapsMapId` so idle symbols receive the same high `z-index` pinning as inspector overlays (above WME Places).
- **Map Inspector / Places (stacking):** OpenLayers 2 sets `layer.div.style.zIndex` without `!important` in `setZIndex` / `resetLayersZIndex`, which overwrote our pinned overlay z-order after redraw or layer reorder—idle ESRI symbols could sit under WME Places while inspector-only layers still looked correct. The script now hooks `OpenLayers.Layer.prototype.setZIndex` and `map.resetLayersZIndex` to re-apply the overlay `!important` z-index values immediately after OL resets them.
- **Map Inspector / Places (stacking, v2026.04.03.18):** Overlay z-order is applied via **`!important` rules in an injected stylesheet** on `.openmaps-ol-overlay-z[data-openmaps-ol-ztok="…"]` (and matching `canvas`/`svg` descendants). That survives OpenLayers assigning plain inline `z-index` (author `!important` in a stylesheet outranks non-important inline). `OpenLayers.Map.prototype.resetLayersZIndex` is hooked as well as the live map instance, and reapplies only for `W.map.getOLMap()` so other OL maps on the page do not rewrite the shared rule block.
- **Map Inspector / Places (stacking, v2026.04.03.19):** Pinning and z-index styling now use **only layers present on `olMap.layers`**, then **sweep** the map for any `OpenMaps_ESRI_FEATURE_*` vector (and OpenMaps bbox vectors marked with `openMapsBboxMapId`) missing from that resolved list. Falling back to the handle’s layer when `indexOf` was −1 caused **`setLayerIndex` to no-op** and stylesheet hooks to target a **stale `div`**, while idle points kept drawing on WME’s live layer — matching “selected/hover OK, idle under Places.” **Google Places** sits above venue fills because WME already orders that native layer higher; it is not the same failure mode as our stale handle ref.
- **Map Inspector / Places + multi-ESRI (v2026.04.03.20):** Map hover/click no longer maps `hit.layer` to a catalog row with `h.layer === hit.layer` (breaks when WME swaps the OpenLayers.Vector instance). Uses `openMapsMapId`, `OpenMaps_ESRI_FEATURE_<id>`, and `openMapsResolveLayerOnOlMap` so stacked symbols from different FeatureServer maps don’t thrash. Swept overlay layers sort by **Active Maps** order so pins don’t reshuffle relative ESRI order every frame. Overlay layer `div`s that are direct children of `viewPortDiv` are **moved to the end** for paint-order tie-breaks; stylesheet rules are prefixed with **`#<olMap.div.id>`** when available. **`moveend`** triggers a debounced re-pin after pan.
- **Map Inspector / Places (v2026.04.03.21):** Pinned OpenMaps/inspector/ESRI overlay layers use **`pointer-events: none`** on the layer `div` and nested `canvas`/`svg`. They still paint above WME Places, but no longer steal the DOM hit target from Places (sibling layer divs do not receive events from another branch’s bubble path). FeatureServer picks still use **document capture** + map-pixel hit-testing. Reduces “Places unselectable until pan” after pointer leaves the map or WME toggles selection.

- **SDK / bootstrap (v2026.04.03.22):** Binds the official WME SDK via `getWmeSdk` (stable GreasyFork script id), registers the sidebar with `Sidebar.registerScriptTab` when possible, listens for map pan/zoom through SDK `Events` with OpenLayers fallbacks, toggles satellite imagery through `Map.setLayerVisibility` when supported, caps the OpenLayers readiness poll (~90s) with a clear console message, and uses Bootstrap 5 tooltips instead of jQuery. Documented polyfills remain for OpenLayers z-index stacking and `GM_xmlhttpRequest` / `unsafeWindow` where CSP or the Tampermonkey sandbox requires them.
- **Structure / maintenance (v2026.04.03.23):** Short region map comment after the Tampermonkey header; `//#region OpenMapsSdkBootstrap` wraps SDK helpers, OpenLayers readiness polling, and the start of `onWmeReady` through the SDK bind/log block (translations and the rest of the script stay outside for mergeability); `//#region OpenMapsBoot` wraps `onWmeInitialized`, `bootstrap`, and the entry call. No behavior change.
